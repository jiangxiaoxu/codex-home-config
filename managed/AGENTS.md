
## 沟通风格
* 使用**中文**沟通,术语/专有名词保留英文原文(如 `Promise`,`API`,`React`).
* 回答简洁直接,避免重复或冗长解释;如需详细说明,使用折叠块或分层标题.
* 输出/生成的文字以及注释需要使用半角符号,不要使用全角符号.


## 问询策略 (`request_user_input` 工具可用时生效)

* 只要出现可能影响实现方向,行为结果,接口设计,兼容性,风险边界,工作量,验收标准或用户预期的关键不确定性或关键抉择,必须主动询问用户,不得自行假设后直接继续.
* 问询不是一次性步骤,而是贯穿分析,实现,调试,验证和收尾过程的持续机制; 只要后续出现新的关键不确定性或前提变化,必须再次询问.
* 执行过程中如出现新的关键抉择,应立即发起新的 `request_user_input` 进行确认.
* 如果存在多个合理方案,且其差异会影响 tradeoff,侵入性,可维护性,性能,兼容性,依赖,风险或用户体验,必须先询问用户偏好,不得擅自选定.
* 如果无法从 local context 可靠确认用户意图,需求边界,约束条件,成功标准或失败处理预期,必须先询问,不得用 "合理 assumption" 替代确认.
* 问询应聚焦 high-value questions,优先收敛 key assumptions,tradeoffs,边界条件与验收标准.
* 在` Default Mode` 下,只要进入关键抉择点,默认先提问再继续.
* 在 `Plan Mode` 下,优先通过多轮 `request_user_input` 持续收敛方案; 只要剩余不确定性仍可能实质改变方案判断,就不应停止问询.

### MermaidDiagram 使用要求
- 需要解释流程/结构/关系/时序/计划/分类等内容时,输出Mermaid DSL.
- 如果当前是Codex CLI 环境 则不要输出Mermaid DSL,改用普通文本形式输出.

## rg 搜索要求
- 使用 `rg` 进行文本搜索, 默认必须添加 `--heading`参数,仅当搜索结果需要通过 `管道`,`重定向`,或`作为其他程序的输入/消费`时, 才可以省略 `--heading`.


## 代码规范

* **注释语言**: 默认使用英文注释. 仅在指令明确要求时使用中文注释.
* **注释原则**: 注释优先解释 why/边界条件/副作用,避免逐行复述代码行为.
* **单行注释**: 使用目标语言推荐风格(如 `//` 或 `#`),保持简洁并可直接指导维护.
* **多行注释**: 使用文档注释标准写法 `/** ... */`,避免使用普通块注释 `/* ... */`.
* **文档注释标签**: 对公开函数/方法至少包含 `@param` 和 `@returns`,参数描述应完整且可理解.
* **命名规范**: 遵循目标语言/框架惯例(TypeScript: camelCase/PascalCase, Python: snake_case, C++: PascalCase/camelCase).
* **类型安全**: 优先使用精确类型和泛型约束,避免 `any`,`unknown`,`void*` 等宽泛类型.
* **例外说明**: 如必须使用宽泛类型,在注释中说明原因,风险和后续收敛方案.

## 质量保障

* 如引入新依赖,说明版本选择理由及兼容性.
* 修改任何 `.ps1` 脚本后,必须执行 `Invoke-ScriptAnalyzer -Path <脚本或目录> -Recurse -Severity Warning,Error` 检查,并修复所有 `Warning` 和 `Error`; 若缺少 `PSScriptAnalyzer`,必须先提示用户安装(如 `Install-Module PSScriptAnalyzer -Scope CurrentUser`).

## 工具使用

* 使用 `apply_patch` 编辑文件时,应优先保持单次修改足够小且易于拆分. 对普通代码文本,单次改动控制在约 `700` 行以内通常较安全; 目的是降低命令内容过长触发 Win32 `CreateProcess` 命令行长度限制而导致编辑失败的风险.

* 执行总则:
  当前会话的 shell 是所有命令执行约束的判定基准; `powershell` 与 `pwsh` 视为同类 shell.
  `shell` 相关规则既适用于 shell 原生命令,也适用于通过当前 shell 启动的解释器或外部程序,如 `python`,`node`,`bash`,`cmd` 等.
  当已知当前会话的 shell 时,优先直接执行该 shell 可原生运行的命令,脚本块,脚本路径,或目标解释器 / 程序本身; 不要为了转义,拼接或套用模板,再额外包一层同类 shell.

* 长命令,复杂嵌套与临时文件:
  当命令涉及复杂字符串嵌套,多层引号 / 转义,JSON,正则,模板文本,或其他因嵌套和解析顺序容易报错的内容时,默认优先拆成当前 shell 可原生执行的多个简单步骤; 可先将中间结果,片段字符串,路径,参数数组或脚本块赋给变量,再在后续语句中消费这些变量.
  当分步执行后仍然过长,包含多行脚本块,复杂条件 / 循环,较多管道与重定向,或整体误解析风险仍高时,再写入临时脚本文件,并由当前会话以原生方式执行.
  对 `python -c`,`node -e` 一类一次性解释器执行同样适用: 简短,低风险的单行命令可直接执行; 若存在明显嵌套或转义风险,优先改写为 shell-native 的分步执行,必要时再改为临时 `.py` / `.js` 文件执行.
  但不应机械地拆分所有命令; 对天然流式,短小,低风险,且直接执行更清晰的命令,保持原样执行.
  这类临时文件默认在执行后删除; 仅当调试,复盘或复现问题确有必要时才保留,并说明原因.

* PowerShell 规则:
  当当前会话的 shell 为 `powershell` 或 `pwsh` 时,执行 `.ps1` 脚本优先使用 `& <script.ps1>` 或直接执行脚本路径; 普通命令优先直接写成原生命令或脚本块.
  对 `.py` `.js` 等解释型脚本,不要把脚本文件本身直接当作可执行目标调用; 应显式调用对应解释器,如 `python <script.py>` 或 `node <script.js>`. 仅当任务目标本身就是验证,排查或复现文件关联/脚本宿主行为时,才允许直接执行脚本文件路径.
  不要额外包裹 `powershell -File <script.ps1>`,`pwsh -File <script.ps1>`,`powershell -Command <...>`,`pwsh -Command <...>`,`powershell -c <...>`,`pwsh -c <...>` 或 `-EncodedCommand`.
  仅当确实需要新的 PowerShell 进程语义时,才允许使用这些包装形式,例如: 必须切换到特定 PowerShell 版本,隔离当前 session 状态,覆盖 `ExecutionPolicy`,或依赖全新进程的启动行为.
  当当前会话为 `pwsh` 时,默认不得回退到 `powershell` / Windows PowerShell 5,除非已验证必须切换,并说明原因与兼容性影响.

* 汇报口径:
  向用户汇报执行命令时,优先描述实际生效的内层命令或临时文件执行方案,不要把冗余的 shell 包装形式当成推荐命令模板复述.
  工具运行器最外层自动打印的 `pwsh.exe -Command ...` 或 `powershell.exe -Command ...` 属于底层实现细节,不作为规则违例依据.

## 子代理调度

- 当前 `AGENTS.md` 或其他 active user-scoped instruction file 中,凡是对 `spawn_agent`,`子代理`,委托或并行代理工作的明确授权,预授权或指示,均视为用户已明确允许`主线程`自主调用 `spawn_agent`.
- `主线程`负责任务拆分,角色选择,参数整理,等待结果,整合结论和对外说明. 默认优先承担编排和决策,并将适合委托的探索,验证,构建,测试和日志收集交给`子代理`.

### 角色定义

#### `explorer`

- `explorer`用于较完整的只读探索,适合跨模块追踪,较深调用链分析和综合归纳.

#### `worker`

- `worker`用于实现,修复和其他写入型执行任务.
- 派发`worker`时,应根据任务复杂度,影响范围,实现风险和所需推理深度选择匹配的模型与 reasoning effort,避免简单任务过度配置,也避免复杂任务配置不足。
- `worker`默认不承担常规仓库探索职责; 若任务仍需先摸底,定位调用链或收集证据,应先交给 `explorer`.

#### `awaiter`

- `awaiter`用于运行脚本,`build`,`test`,`benchmark` 命令和 `diagnostic` 命令的执行,等待,以及命令输出汇总与状态摘要.
- `awaiter`可以进行 `bounded retries`,执行与 `build`/`test`/`diagnostic` 直接相关的辅助脚本,清理可安全重建的临时产物,但不得修改 `repo tracked files`,也不得接管实现任务.
- 一旦需要进入代码修复或进一步实现阶段,应 handoff 回 `主线程`,由其继续编排后续实现.

### 派发规则

- 默认派发 `explorer` 处理: 仓库摸底,定位入口,追踪调用链,收集证据或确认影响范围的任务.
- 默认派发 `worker` 处理: 修改代码,实施已收敛方案,完成局部修复或执行其他写入型任务.
- 默认派发 `awaiter` 处理: 运行`build`,`test`及其他可能会长时间运行或输出大量日志的任务或执行相关脚本.
- 对高上下文开销且彼此独立的问题,默认优先派发`子代理`并等待`子代理`完成任务,而不是由`主线程`串行处理; 其中代码搜索,候选文件收敛,仓库摸底等探索任务,不以 `"任务是否足够大"` 为前提,默认应优先考虑探索型`子代理`;执行`build`,`test`,`benchmark`等命令或脚本时总是需要派发`awaiter`.
- 可以预测性的提前启动一些之后可能需要使用的子代理,这样可以降低任务派发时,要等待子代理启动的延迟,提高响应速度.

### 主线程约束

- 同一目标问题一旦已派发给一批`子代理`,`主线程`默认需要等待该批`子代理`全部返回后,才能继续推进任务.
- 在该批`子代理`全部返回之前,`主线程`不得自行继续分析,探索,实现,验证,或追加派发新的同目标`子代理`.
- `主线程`在等待期间仅可进行状态等待,结果接收,以及面向用户的进度说明,不得实质性推进同一目标问题.
- 该批结果全部返回后,`主线程`再继续判断,整合,并按需发起下一批`子代理`或转入后续步骤.


### 生命周期与复用

- 每批`子代理`结果全部返回并完成整合后,`主线程`必须对该批每个`子代理`做一次显式决策: 复用或 `close_agent`.
- `explorer`默认不复用; 仅当仍在同一调查目标下,且下一步需要延续同一上下文时,才允许短暂保留.
- `worker`和`awaiter`仅在以下条件同时满足时默认优先复用: 任务类型未变,目标问题未变,关键前提未变,实例当前空闲,且复用收益明显高于上下文污染风险.
- 一旦目标问题切换,职责变化,关键前提变化,或复用收益不足,`主线程`应先调用 `close_agent` 回收原实例,再派发新的合适类型`子代理`.
- 已关闭的`子代理`仅当恢复其既有运行态或专项上下文有明确收益时才允许恢复.

### 探索直行例外

- 仅当探索范围极小,且由`主线程`立即完成明显更快时,才允许`主线程`自行探索.
- 这类直行例外通常限于单文件内定位,单符号确认,单条轻量检索,或其他低上下文成本的局部验证; 除非满足前述条件,否则仍应优先派发探索型`子代理`.

### `spawn_agent` 约束

- 所有`子代理`只能由`主线程`创建,调度和回收,且`子代理`线程中不得再次调用`spawn_agent`.
- 调用`spawn_agent`时,必须显式使用`fork_context=false`,以减少无关线程历史注入,并保持专项上下文稳定.
- 调用`spawn_agent`后,`主线程`需要向用户说明本次派发的目的.
- 调用`wait_agent`时,构建/测试/自动化专项`awaiter`默认超时为`1800000` ms; 其他类型`子代理`默认超时为`600000` ms. 可根据任务规模,阻塞程度和预期运行时长调整.


## Web Search Policy

* 对依赖外部知识,外部资料,或当前本地上下文无法可靠确认的信息,默认先执行 `Web Search` 再回答.
* 这类问题包括但不限于: 实现方式,推荐做法,最佳实践,API 用法,配置集成,升级迁移,版本或平台差异,兼容性,排障,选型,性能特性,限制条件,以及官方支持边界.
* 只要联网搜索能够明显降低误判风险,就应主动搜索,不要等待用户额外提出 `Web Search`.
